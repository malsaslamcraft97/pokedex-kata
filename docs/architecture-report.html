What Nik Sumeiko's model prescribes

Principle 01
Separate domain from application logic
Domain logic — pure functions, no React, no side effects. Filtering a Pokémon list by type is a business rule; it should work in any environment. Application logic — React hooks that wire domain functions into the component lifecycle. It knows about useState and useEffect, but not about HTTP.
Principle 02
Dependencies point inward
UI imports from the application layer. The application layer imports from domain. Domain imports nothing from React or the network. This is Dependency Inversion: outer layers know about inner ones, never the reverse. Breaking this rule is what makes refactoring expensive.
Principle 03
Outside-in TDD
Write an acceptance test (component level) first. Watch it fail. Then drive the domain functions into existence with plain unit tests. Pure domain functions can be tested with a single expect(fn(input)).toBe(output) — no RTL, no MSW, no store.
Principle 04
Black-box testing of domain
Test domain functions through their public contract, not their implementation. filterByType(list, ['fire']) should return only fire-type Pokémon — the test doesn't care how the function is written, only what it returns. This makes internal refactoring safe.

const processedList = useMemo(() => {
  let list = [...data];

  // filterBySearch — pure business rule
  if (searchText)

  // filterByType — pure business rule
  if (filters.type) {
    const selectedTypes = filters.type.split(",");
    list = list.filter(p => p.types.some(t => selectedTypes.includes(t.toLowerCase())));
  }

  // sortBy* — pure business rule
  switch (filters.sortBy) {
    case "name_asc": list.sort((a, b) => a.name.localeCompare(b.name)); break;
    // …3 more cases
  }
}, [data, searchText, filters]);
This logic has no dependency on React whatsoever. Wrapping it in useMemo is a performance optimisation, not an architectural requirement. Extract to domain/pokemonFilters.ts and domain/pokemonSorters.ts, then call them from the hook.

Transformation plan — specific files and actions

Create src/features/pokemon/domain/pokemonFilters.ts
Extract the filter logic from usePokemonListPageData.ts (lines 67–84) into pure functions.
  • filterBySearch(list: EnrichedPokemon[], search: string): EnrichedPokemon[]
  • filterByType(list: EnrichedPokemon[], types: string[]): EnrichedPokemon[]
  • filterByAbility(list: EnrichedPokemon[], ability: string): EnrichedPokemon[]
Zero imports from React or Redux. Testable as: expect(filterByType(mockList, ['fire'])).toHaveLength(1).
Create src/features/pokemon/domain/pokemonSorters.ts
Extract the sort switch-case from usePokemonListPageData.ts (lines 88–113).
  • sortByHeight(list, direction: 'asc' | 'desc'): EnrichedPokemon[]
  • sortById(list, direction: 'asc' | 'desc'): EnrichedPokemon[]
  • sortByName(list, direction: 'asc' | 'desc'): EnrichedPokemon[]
  • applySort(list, sortBy: string): EnrichedPokemon[] — orchestrates the above based on the sortBy string
Create src/features/pokemon/domain/pokemonTransformers.ts
Move the inline enrichment block from pokemonApi.ts (lines 53–71) into a pure function.
  • enrichPokemon(raw: RawPokemonDetail): EnrichedPokemon — formats ID, capitalises name and types, selects image, extracts ability
pokemonApi.ts then calls results.map(enrichPokemon) — one line instead of twenty. The type EnrichedPokemon can be moved here too, breaking the UI's direct import from api/pokemonApi.
Split hooks/usePokemonListPageData.ts → 3 focused hooks
The 172-line hook becomes three independently testable hooks:
  • usePokemonListData.ts — RTK Query + calls domain filter/sort functions, returns processedList. ~40 lines.
  • useInfiniteScroll.ts — only the IntersectionObserver setup. Takes an onLoadMore callback, returns loadMoreRef. ~30 lines.
  • useUrlFilterSync.ts — only the URL ↔ Redux bidirectional sync. Runs on mount and filter change. ~35 lines.
PokemonListPage composes all three. Each hook now has a single, mockable responsibility.
Update api/pokemonApi.ts
Replace the inline enrichment block (lines 44–73) with a call to enrichPokemon from the domain layer. The endpoint's only job becomes: fetch the list, fetch each detail in parallel, call enrichPokemon, return. All formatting rules live in domain/pokemonTransformers.ts — tested separately, independently of the network.
Remove store/pokemonSlice.ts
pokemonSlice defines three actions (setLoading, setPokemon, setError) that are never dispatched anywhere in the codebase. RTK Query already manages loading and error state via its own cache. Delete the slice and remove it from app/store.ts. This removes dead code and makes the state surface smaller.

Is the current architecture better than Nik's model in any way? Yes — one specific thing. The feature-based folder structure (features/pokemon/) is more pragmatic than a global layered structure for a single-feature app. Nik's model doesn't prescribe feature folders at all; you combined feature encapsulation with concern layering, which is architecturally ahead of pure global layering. That structure stays intact — the transformation adds a domain/ subfolder inside it.

What the transformation unlocks: Once the domain layer exists, the filtering and sorting logic can be tested as plain functions in milliseconds — no RTL, no Redux store, no MSW server. The hooks become thin wires connecting domain logic to React's lifecycle. The API becomes a fetch contract with no formatting opinions. The components stay exactly as they are. Three new files, one split hook, one deleted slice. Smallest surface area change for the largest architectural clarity gain.
